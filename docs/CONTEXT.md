# Glossary

Canonical terms for the sound matching evaluation framework. This file defines language
only — implementation decisions live in `DECISIONS.md`, architecture in `ARCHITECTURE.md`.

| Term | Definition |
|---|---|
| **Sound matching** | Predicting the parameter settings of a target synthesizer such that its rendered audio matches a target waveform. Synonym in the literature: inverse synthesis, synthesizer inversion. |
| **Preset** | A complete parameter configuration for a synthesizer. Synonym: patch — the two terms are interchangeable in this project, which prefers *preset*. |
| **Patch** | Same as preset. |
| **Parameter name** | The plugin-reported string identifying a parameter (e.g. `ALGORITHM`, `OP1 OUTPUT LEVEL`). The only way parameters are referred to above the wrapper. |
| **Synth-side dict** | A `dict[str, float]` mapping parameter names to normalized floats in [0,1] — the format the synthesizer wrapper accepts and returns. Categorical values are grid points on [0,1]. |
| **ML-side vector** | A flat `np.ndarray` consumed/produced by models: continuous params as floats, categorical params expanded to one-hot blocks. |
| **ParameterSpace** | The abstraction owning the ordered parameter subset definition and the two-way conversion between synth-side dict and ML-side vector. |
| **ParameterSpecification** | The specification of a single estimated parameter: its name, kind, bounds or options, and default. A ParameterSpace is an ordered list of ParameterSpecifications. |
| **Kind** | Whether a parameter is *continuous* (perceptually smooth scale, regressed as a float) or *categorical* (unordered classes or perceptually discontinuous grid, predicted as a class). |
| **One-hot block** | The contiguous span of an ML-side vector representing one categorical parameter: one slot per option, exactly one set to 1. |
| **Loss slices** | The partition of the ML-side vector into per-parameter spans, each tagged with its kind, used to route losses and parameter-side metrics. |
| **D1 subset** | The final 103-parameter set the models estimate, locked by D1 (see `docs/DECISIONS.md`): preset-gen-vae's full DX7 voice minus the keyboard-scaling and velocity params that are non-identifiable under the fixed C4 / velocity-100 render. Defined in `synth/dexed/subset.py`. |
| **Subset** | The chosen set of parameters that models estimate. All non-subset parameters are *locked* at defaults. All benchmarked models target the same subset. |
| **Locked parameter** | A parameter held at its default value and never estimated, randomized, or exposed in the dataset. |
| **Excluded parameter** | A VST-level parameter (e.g. `Bypass`, `Program`, MIDI CC passthroughs) that the wrapper never exposes at all. Stronger than locked. |
| **Render contract** | The guarantee that rendering the same synth-side dict in the same render context yields bit-identical audio. Context-independence (same dict, any prior render history) is not guaranteed by the synthesizer. |
| **Render context** | The complete in-process history (instance creation and prior renders) preceding a render. Renders are deterministic given an identical context. |
| **Context leakage** | The within-engine shift of a patch's rendered audio caused purely by which patches were rendered before it (i.e. by the render context), via Dexed's hidden per-voice state. Concentrated in LFO / sample-&-hold / noise voices; quantified by `scripts/measure_context_leakage.py`. See D-REPRO. |
| **Renderer** | The engine-specific layer beneath the synthesizer wrapper that hides one VST host (DawDreamer default, Pedalboard secondary). Selected via `DexedWrapper(renderer=...)`; chosen once per run and recorded in metadata (renderers are never mixed — D-RENDERER). |
| **Reuse arm** | A benchmarking strategy in which one persistent wrapper renders every patch in sequence — the framework's default render path. Contrast with the reload arm. |
| **Reload arm** | A benchmarking strategy (a.k.a. *reload-per-render*) in which a fresh wrapper — rebuilt engine and plugin — is constructed for every render, mirroring preset-gen-vae's approach. *In-process*: the new engine is built in the same OS process, so it may inherit prior heap state and is not guaranteed a clean render context (cf. **Render context**, D-REPRO). |
| **Render duration** | Total length of rendered audio (4.0 s). |
| **Note duration** | Time from note-on to note-off within the render (3.0 s); the remainder captures the release tail. |
| **Synthetic corpus** | A training corpus built by the *synthetic* construction method: uniform random subset presets (over the audible region), each rendered. One arm of the construction-method axis. (Formerly "synthetic dataset".) |
| **Human preset test set** | The shared held-out evaluation corpus: the *test* partition of deduplicated human presets (e.g. DX7 SysEx cartridges), testing generalization to musically realistic sounds. The *train* partition of the same presets feeds the human and hybrid training corpora — see **Partition**, **Construction method**, and D4 in `DECISIONS.md`. |
| **Model family** | A top-level modeling approach entered in the benchmark as one registered representative (e.g. discriminative — Sound2Synth lineage; generative — preset-gen-vae lineage). Drawn from the six SLR taxonomy categories; the final benchmarked set is D-FAMILIES (open) in `DECISIONS.md`, not all six. |
| **Metric panel** | The fixed set of evaluation metrics computed for every model on every test sample (parameter-side diagnostics + audio-side spectral/perceptual distances). Organized as independent per-sample metrics grouped by **metric axis**, deliberately over-provisioned and pruned to a representative subset using rank-correlation across samples. |
| **Metric axis** | The aspect of similarity a metric captures: magnitude-spectrum, timbral envelope, perceptual (embedding), loudness/dynamics, pitch, or parameter-space. Metrics are grouped by axis so that near-collinear metrics within an axis can be pruned to a representative few. The perceptual (embedding) axis is defined but **deferred — not implemented in this thesis** (see `D-METRIC-PERCEPTUAL` in `DECISIONS.md`). |
| **Re-render** | The Evaluator step that renders a model's predicted synth-side dict (as `default_params ⊕ prediction`) back into audio, to compare against the stored target WAV. Done **fresh-process at sequence position 0**, identically to how the eval corpus's targets were rendered, so target and prediction share a clean context (D-REPRO). The target is never re-rendered. |
| **Evaluator** | The Layer-4 consumer of the **metric panel** (`evaluation/evaluator.py`): given a fitted model and a loaded corpus, it predicts, **re-renders**, runs every panel metric, aggregates, and writes an **eval run**. Reads its render contract from the corpus's **run summary**, never `config.py` (D-EVAL). |
| **Eval run** | One Evaluator execution of one model on one corpus, persisted as a self-describing folder `results/<corpus_name>/<model_name>/` (parallel to `dataset/<run_name>/`): a **per-sample matrix** (`per_sample.csv`) plus an `eval_summary.json` (render contract echoed from the corpus, **checkpoint fingerprint**, per-metric mean/std/valid-count). |
| **Per-sample matrix** | The N-samples × M-metrics table of raw metric values (`per_sample.csv`), `NaN`s intact. The source of truth for the metric-panel pruning analysis (rank-correlation across samples); aggregates are derived from it as `nanmean` + std + valid-count. |
| **Valid-count** | Per metric, the number of samples whose value is non-`NaN` (the metric was defined there). Reported next to every aggregate so an "undefined hides failure" case (e.g. silent predictions making a pitch metric undefined everywhere) stays visible, not masked. A `NaN` means *metric undefined for this sample* — not zero, not error. |
| **Checkpoint fingerprint** | The sha256 hash of the model checkpoint file an **eval run** scored, recorded in `eval_summary.json` so a results folder is traceable to the exact weights. A null hash with a note is recorded when the model had no file (fitted in-memory). |
| **Construction method** | How a corpus's presets are produced: *synthetic*, *human*, or *hybrid*. Recorded per preset (provenance) and per run (run summary). Treated as a planned experimental axis — models are trained separately per method and evaluated on one shared human test set; see D4 in `DECISIONS.md`. |
| **Preset source** | The synth-agnostic abstraction (`dataset/preset_sources.py`, `PresetSource`) that yields a deterministic, finite stream of presets to render. Implementations: `SyntheticPresetSource`, `HumanPresetSource`, `HybridPresetSource`. Randomness derives from a master seed via per-slot streams, so the stream is reproducible and order-independent. |
| **Preset record** | One item from a preset source (`PresetRecord`): a subset synth-side dict plus provenance (method, partition, source file / voice, parent). The unit the **dataset builder** renders into one sound and one metadata row. |
| **Synthetic preset source** | The preset source for the synthetic method (`SyntheticPresetSource`): uniform random subset presets; near-silent draws are redrawn until audible (see **Near-silence redraw**). |
| **Human preset source** | The preset source for the human method (`HumanPresetSource`): human-made presets projected onto the subset (see **Subset projection**), tagged with their partition. |
| **Hybrid preset source** | The preset source for the hybrid method (`HybridPresetSource`): combines human-*train* presets with synthetic material by **blend** or **augmentation**. Derives only from the human train partition, never the held-out test set. A future distribution-sampling mode is left as a slot. |
| **Blend** | Hybrid construction that interleaves a synthetic stream with human-train presets at a fixed ratio; each output preset keeps its own provenance (synthetic or human). |
| **Augmentation** | Hybrid construction that perturbs a human-train preset — jitter continuous params, optionally flip a categorical — into a new, unseen preset tagged with its parent (`parent_id`). |
| **Subset projection** | Keeping only a preset's subset parameters and letting the dropped parameters fall back to synth defaults at render time. Near-lossless under the fixed render contract (the dropped params are non-identifiable there — D1), so it lets every preset source emit a uniform subset dict the builder renders as `defaults ⊕ subset`. |
| **Preset loader** | The synth-specific component that normalizes on-disk presets into a flat, deduplicated, train/test-split stream (e.g. `dataset/dexed_preset_loader.py`, `DexedPresetLoader`: load cartridge voices → dedup → voice-level split). The only synth-dependent piece of the human/hybrid path. |
| **Deduplication** | A load-bearing loader pass dropping exact and near-duplicate presets — compared on their **subset projection**, so twins differing only in dropped params collapse — before the split, so no near-twin straddles train and test. |
| **Partition** | The train/test tag every preset carries. The test partition is always human (the shared held-out set); the train partition may be synthetic, human, or hybrid. Assigned by the loader's seeded voice-level split. |
| **Corpus split** | Splitting an *already-rendered* corpus into a train corpus and a test corpus (`scripts/split_corpus.py`, `dataset/corpus_splitter.py`), distinct from the build-time **Preset loader** split (which splits presets before rendering). The train partition's audio is copied; the test partition is re-rendered fresh-process at pos 0 (D-REPRO) so it satisfies the eval contract. A seeded row-partition (same `split_indices` as the loader), no re-**deduplication**, and **hybrid corpora are refused** (train/test leakage). See D-SPLIT in `DECISIONS.md`. |
| **Train/test leakage** | An ML data-leakage risk distinct from render **Context leakage**: a preset and its near-duplicates or derivatives straddling the train and test partitions, which inflates measured performance. Guarded by **Deduplication** at build time (no near-twin straddles the split) and by excluding hybrid corpora from a **Corpus split** (augmented children / repeated blend parents share human parents). |
| **Dataset builder** | The synth-agnostic component (`dataset/builder.py`, `DatasetBuilder`) that renders a preset source into a corpus: for each preset, render `defaults ⊕ subset` under the render contract and write a WAV plus a metadata row, plus a run **summary**. Reproducible from the source's seed. |
| **Render executor** | The pluggable strategy the dataset builder uses to render presets (`RenderExecutor`), decoupling isolation policy from corpus logic: the in-process `SequentialExecutor` now; `spawn` worker pools (training) and fresh-process-per-render (test) later — see D-REPRO. |
| **Near-silence redraw** | Builder policy: a preset whose rendered peak amplitude is below a threshold is redrawn (generative sources only) up to a retry cap; the `near_silent` flag is recorded regardless. Human presets are kept and flagged, never redrawn. |
| **Run summary** | The per-run `run_summary.json`: render settings, renderer, seeds, subset, default-parameter vector, the serialized **ParameterSpace** (so the corpus is self-describing — the ML-side target vector can be rebuilt offline with no live synth/VST; see D-SELFDESC), source description, and git revision — everything needed to reproduce a corpus. Paired with `metadata.csv` (one row per sound) and `audio/<id>.wav`. |
| **Training job** | One SLURM job submitted by the dashboard's cluster page: a corpus, a model family, and a training config, run on the PUT cluster via `cluster/train.sbatch`. Tracked locally by its SLURM job id, submission time, and current state (pending/running/completed/failed/cancelled) — see **Job registry**. |
| **Job registry** | The local, gitignored `cluster/jobs.json` recording every **training job** the dashboard has submitted, so the dashboard can list and poll jobs across restarts without querying the cluster for history it doesn't keep. |
